## Stack

- **MkDocs Material** — config in `mkdocs.yml`, served with `hatch run docs:serve`.
- **MkDocs hooks** in `mkdocs-hooks/` — registered in `mkdocs.yml` under `hooks:`.
- **docs-build** CI workflow runs on all PRs (strict mode) and is in the CI gate.

## Design Lifecycle

For non-trivial changes, this optional workflow moves ideas through to
implementation:

```
idea/problem → explorations/
proposed design → blueprints/
decision locked in → adr/
build steps → implementation-plans/
```

Skip stages that don't apply. Each directory has a README explaining when
to write that type of document.

## Conventions

- Explorations go in `docs/explorations/` — early-stage idea evaluation.
- Blueprints go in `docs/blueprints/` — proposed structural designs.
- ADRs go in `docs/adr/` (see `docs/adr/.instructions.md` for detailed procedure).
- Implementation plans go in `docs/implementation-plans/` — step-by-step build details.
- Design docs go in `docs/design/`.
- Reference material goes in `docs/reference/`.
- Use relative links between docs. The `repo_links.py` MkDocs hook rewrites repo-relative links to GitHub URLs at build time.
- Don't write marketing copy. State what something does, its limitations, and move on.
