# ADR Instructions

Rules for Architecture Decision Records in `docs/adr/`.

## Adding a New ADR

1. **Find the next number.** Look at existing files in `docs/adr/` and pick the next sequential `NNN`.
2. **Copy the template.** Copy `docs/adr/template.md` to `docs/adr/NNN-short-title.md`.
3. **Fill in all sections.** At minimum: Status, Context, Decision, Alternatives Considered, Consequences, Implementation, References.
4. **Update the index.** Add the new ADR to `docs/adr/README.md`.
5. **Update copilot-instructions.md.** If the ADR affects day-to-day work (like ADRs 001, 008, 024, 031), add it to the "Key ADRs" table in `.github/copilot-instructions.md`.
6. **Update architecture docs.** If the decision touches architecture or tooling, update `docs/design/architecture.md` and/or `docs/design/tool-decisions.md`.

## Conventions

- Use **imperative title**: "Use X for Y", not "We decided to use X".
- Status is one of: `Proposed`, `Accepted`, `Deprecated`, `Superseded by [ADR XXX](XXX-name.md)`.
- Link related ADRs in the References section.
- Include implementation links (relative paths) so future readers can find the code.
- Reserve ADRs for decisions that are hard to reverse, affect multiple parts of the project, or would be useful context for future contributors. Not every change needs one.
