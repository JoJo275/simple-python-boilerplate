This directory uses the **src-layout** pattern for Python packaging.

## Why src/?

The `src/` directory is a container that:

- Prevents accidental imports of uninstalled code
- Forces you to install the package (`pip install -e .` or `hatch shell`)
- Ensures tests run against the installed package, not local files

## Structure

```text
src/
в””в”Җв”Җ simple_python_boilerplate/   # вҶҗ The actual package
    в”ңв”Җв”Җ __init__.py
    в”ңв”Җв”Җ main.py
    в”ңв”Җв”Җ cli.py
    в”ңв”Җв”Җ engine.py
    в”ңв”Җв”Җ api.py
    в”ңв”Җв”Җ _version.py              # Auto-generated by hatch-vcs
    в”ңв”Җв”Җ py.typed                 # PEP 561 type-checker marker
    в”ңв”Җв”Җ dev_tools/               # Developer utilities (shipped with package)
    в”ңв”Җв”Җ sql/                     # Embedded SQL queries (shipped with package)
    в””в”Җв”Җ ...
```

Each module in the package has a specific responsibility. The layering
keeps concerns separated so entry points stay thin and core logic
stays interface-agnostic:

```text
  Entry points (pyproject.toml)
        в”Ӯ
        в–ј
    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
    в”Ӯ  main.py в”Ӯ   Thin wrappers вҖ” dispatches to cli / engine
    в””в”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”ҳ
         в”Ӯ
    в”Ңв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”җ
    в”Ӯ          в”Ӯ
    в–ј          в–ј
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ cli.py в”Ӯ в”Ӯ api.py в”Ӯ   Interface layers (CLI, HTTP)
в””в”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”ҳ
    в”Ӯ          в”Ӯ
    в–ј          в–ј
  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
  в”Ӯ  engine.py   в”Ӯ   Core logic вҖ” no I/O, no framework deps
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

| Module         | Role                                                                                              |
| -------------- | ------------------------------------------------------------------------------------------------- |
| `__init__.py`  | Package root. Exports `__version__` (from `_version.py` with release-please fallback).            |
| `main.py`      | Entry-point wrappers: `main()`, `print_version()`, `doctor()`. Delegates immediately вҖ” no logic.  |
| `cli.py`       | CLI contract: `argparse` parser, subcommands, flags. Calls `engine` for actual work.              |
| `api.py`       | HTTP/REST interface placeholder. `create_app()` factory вҖ” add FastAPI/Flask routes here.          |
| `engine.py`    | Core business logic. Pure functions, TypedDicts, no I/O. Both `cli.py` and `api.py` call into it. |
| `_version.py`  | Auto-generated by `hatch-vcs` from git tags. **Do not edit** вҖ” gitignored.                        |
| `py.typed`     | PEP 561 marker file. Tells type checkers this package ships inline types.                         |
| `dev_tools/`   | Developer utility modules shipped with the package.                                               |
| `sql/`         | Embedded SQL queries (e.g., `example_query.sql`). Shipped with the package via `include`.         |

**Key principle:** `engine.py` never imports from `cli.py` or `api.py`.
Data flows _inward_ вҖ” interface layers call engine, never the reverse.
This makes it easy to add new interfaces (e.g., a gRPC layer) without
touching core logic.

## Usage

```bash
# Preferred: Use Hatch for development
hatch shell

# Alternative: Install in editable mode
pip install -e .

# Then import normally
python -c "from simple_python_boilerplate import main"
```
