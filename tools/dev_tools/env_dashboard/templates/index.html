Warnings ({{ warnings|length }}) Issues detected in your development environment. Click any warning to expand details and recommended fixes. Red = critical (fix now), amber = should fix, blue = informational.

Guide

Automated checks that scan your development environment for problems. Warnings surface before they become hard-to-debug build or runtime failures.

Severity levels
Critical (red ×)
Must fix — will cause builds, installs, or tests to fail. Examples: disk full, insecure PATH, unwritable temp directory.
Warning (amber !)
Should fix — won't break things immediately but indicates problems. Examples: no virtualenv, dirty git tree, dead PATH entries.
Info (blue i)
Awareness only — nothing to fix, but worth knowing. Examples: CI environment detected, WSL detected, PATH duplicates.
All checks performed
  • Virtualenv active — verifies you're running inside a venv (not system Python).
  • Git clean state — flags uncommitted changes that make builds non-reproducible.
  • PATH dead entries — directories listed in PATH that no longer exist on disk.
  • PATH duplicates — same directory appearing more than once (after normalizing case/slashes).
  • Secret env vars — environment variables matching patterns like API_KEY, TOKEN, SECRET.
  • Insecure PATH dirs — world-writable directories in PATH (malicious executable injection risk).
  • SSH keys in env — private key paths stored in environment variables.
  • Disk space >80% — warns at 80%, critical at 90% (pip/builds will fail).
  • Temp dir writable — verifies the system temp directory is writable.
  • DNS resolution — checks if pypi.org resolves (needed for pip install).
  • HTTPS outbound — verifies outbound HTTPS isn't blocked by proxy/firewall.
  • CI detection — identifies CI environments (GitHub Actions, GitLab CI, etc.).
  • WSL detection — flags when running under Windows Subsystem for Linux.
  • Python version match — checks if Python version satisfies the project's requires-python.
  • Outdated pip — flags when pip itself is significantly out of date.
  • Missing critical tools — checks for git, hatch, and pre-commit availability.
When checks are performed
  • On page load — all checks run automatically when the dashboard starts.
  • On refresh — clicking the Refresh button or triggering a scan re-runs all checks with fresh data.
  • Cache window — results are cached for 30 seconds to avoid redundant work. After 30s the next request collects fresh data.
  • Cross-section analysis — the insights engine runs after all other collectors finish, analysing their combined output to derive warnings.
